## 1. What v0.7.0 actually shipped

Reconstructed from commit history and direct verification, not from memory of intent:

**Sprint 3 — Authentication foundation** (`8f68696`)
Supabase Auth via `@supabase/ssr`. Sign up / log in / log out, session-cookie
handling split across `lib/supabase/{client,server,middleware}.ts`, route
protection via the root proxy file (`middleware.ts` → renamed to `proxy.ts`
this release for Next.js 16 compliance).

**Sprint 4 — Project & generation persistence** (`96e71f9`, `4f6b089`, `e7f18da`)
`lib/repositories/{project,generation}.repository.ts` — RLS-backed CRUD over
Supabase, no application-layer ownership filtering (RLS is the sole
enforcement layer, verified sound). `lib/actions/project.actions.ts` for the
Server Action surface. Drizzle package scripts for local migration tooling.

**Sprint 5 — Canonical AST & compiler pipeline** (`dc35f7b`)
The core v0.7.0 deliverable: replaced "Gemini generates SQL/Drizzle/JSON/docs/
Mermaid directly" with "Gemini generates only a `CanonicalSchemaAST`,
everything else is compiled from it deterministically." Full pipeline:
`lib/ast/` (shape + semantic validation), `lib/compiler/` (5 independent
compilers), `lib/ai/providers/gemini.ts` (AST-only provider), orchestrated by
`lib/services/generation.service.ts`. Legacy direct-generation files
(`lib/ai/generate.ts`, `parse-response.ts`, `prompts.ts`) deleted outright.

**Post-release stabilization** (this audit, folded into `4ea2091` + live
Vercel changes, not further code commits)
- Fixed a real correctness bug: Gemini's structured output didn't reliably
  respect the `default.kind` discriminated union (invented `"static"` instead
  of `"literal"`) — fixed via explicit prompt guidance in
  `lib/ai/providers/gemini-prompts.ts`.
- Added `UNSAFE_EXPRESSION` semantic-analysis check (`lib/ast/analyzer.ts`) —
  closes a real SQL-injection-into-generated-artifacts path via
  `expression`/`check` fields.
- Added a 4000-char cap on AI prompts (`lib/actions/generate-schema.ts`) —
  previously only empty prompts were rejected.
- Root cause of "production fails after ~28s / after key rotation": diagnosed
  through five rounds of live investigation as **`gemini-3.5-flash` model
  overload/unavailability plus an incompatible `thinkingConfig` parameter**,
  not quota, not our code's request logic. Fixed by moving to the
  `gemini-flash-latest` alias and removing `thinkingConfig`.
- Production deployment fully re-verified end-to-end after the fix: real
  browser session, real "Library Management System" generation, real
  `200` Server Action response (previously `503`), all 5 artifacts rendering.

## 3. The full request flow, stage by stage

```
UI (components/dashboard/schema-generator.tsx)
   │  user types a prompt, clicks Generate
   ▼
Server Action (lib/actions/generate-schema.ts :: generateSchema)
   │  requireUser() — auth gate
   │  zod: prompt.trim().min(1).max(4000), projectId.uuid()
   │  on failure → { status: "INVALID_INPUT" }
   ▼
Generation Service (lib/services/generation.service.ts :: generateAndPersistSchema)
   │  getProjectById(projectId) — RLS-gated ownership check, before the
   │    expensive AI call. Failure → PROJECT_NOT_FOUND
   ▼
AI Provider (lib/ai/providers/gemini.ts :: geminiProvider.generateAST)
   │  genAI.models.generateContent({
   │    model: aiConfig.model ("gemini-flash-latest"),
   │    systemInstruction: AST_SYSTEM_PROMPT,
   │    responseMimeType: "application/json",
   │    responseJsonSchema: z.toJSONSchema(canonicalSchemaASTSchema),
   │    httpOptions: { timeout: 30_000 },
   │  })
   │  parseAstResponse(): checks blockReason / finishReason / JSON-parseability
   │    — envelope-level only, NOT shape validation
   │  failure → AI_ERROR (mapGeminiError: 401/403 → "misconfigured", 429 →
   │    "too many requests", else generic; AbortError → "timed out")
   ▼
Canonical AST (unknown JSON at this point, typed CanonicalSchemaAST optimistically)
   ▼
Validation (lib/ast/validator.ts :: validateASTShape)
   │  canonicalSchemaASTSchema.safeParse() — Zod structural check
   │  + astVersion ∈ SUPPORTED_AST_VERSIONS check
   │  failure → INVALID_AST
   ▼
Analyzer (lib/ast/analyzer.ts :: analyzeSchema)
   │  pure, deterministic, never throws. Checks: duplicate table/column,
   │  FK source/target table/column existence, FK column-count match, index
   │  column existence, PK column existence/conflict, UNSAFE_EXPRESSION
   │  (errors) + missing PK / reserved keyword / circular FK (warnings)
   │  failure (errors.length > 0) → SEMANTIC_VALIDATION_FAILED
   ▼
Compiler Registry (lib/compiler/index.ts :: createCompilerRegistry().compileAll)
   │  runs all 5 registered compilers against the same AST; independent
   │  failures don't block each other (CompilerResult<TOutput>, not throws)
   ├──▶ SQL       (lib/compiler/sql)       — reference implementation
   ├──▶ Drizzle   (lib/compiler/drizzle)   — matches project's own schema.ts conventions
   ├──▶ JSON      (lib/compiler/json)      — 3 deterministic sample rows/table
   ├──▶ Documentation (lib/compiler/markdown) — ToC + per-table sections + relationships
   └──▶ Mermaid   (lib/compiler/mermaid)   — erDiagram with inferred cardinality
   ▼
buildGeneratedArtifacts (lib/services/generation.service.ts)
   │  maps CompileAllResult[] → GeneratedSchema. SQL/Drizzle/JSON are
   │  required (compiler failure here → COMPILER_FAILED); Documentation/
   │  Mermaid are optional (failure degrades gracefully, field just omitted)
   ▼
Persistence (lib/repositories/generation.repository.ts :: createGeneration)
   │  Supabase insert, RLS-scoped to the authenticated user via the
   │  project's ownership chain (generations has no user_id column itself;
   │  RLS policies join through projects.user_id)
   │  failure → GENERATED_NOT_SAVED (data still returned to the UI, just
   │    flagged as unsaved via a toast)
   ▼
UI Rendering (components/dashboard/{output-tabs,code-viewer,markdown-viewer,mermaid-viewer}.tsx)
      SUCCESS / GENERATED_NOT_SAVED → render all 5 tabs from `outcome.data`
      everything else → render `outcome.error` as inline text + toast
```

Five distinct failure statuses (`AI_ERROR`, `INVALID_AST`,
`SEMANTIC_VALIDATION_FAILED`, `COMPILER_FAILED`, `GENERATED_NOT_SAVED`) plus
`PROJECT_NOT_FOUND`/`INVALID_INPUT` give the UI enough information to
distinguish failure causes, though the UI currently only discriminates
`SUCCESS`/`GENERATED_NOT_SAVED` (has data) from everything else (has an
error string) — it doesn't yet show *which* stage failed differently.

## 6. Prioritization: Critical / High / Medium / Low

**Critical** (user-facing breakage or data-integrity risk)
1. Project selector shows raw UUID instead of title — visibly broken UX, trivial fix, blocks nothing else.
2. Join tables can silently allow duplicate relationship rows — a real data-integrity gap in every generated schema that has a many-to-many relationship.
3. No FK-column indexing — every generated schema ships with a standard, well-known performance defect.

**High** (blocks confident iteration or is a completed-but-unshipped feature)
4. No automated test suite for the compiler pipeline — every future compiler change is currently unverifiable except by hand.
5. No Git↔Vercel integration — deployment currently depends on someone remembering to run `vercel --prod`.
6. History/navigation UI — repository layer is done, nothing surfaces it; a real feature gap, not just polish.

**Medium** (real quality gaps, not urgent)
7. Native `ENUM` type reconsideration.
8. CHECK-constraint prompt guidance.
9. Composite FK physical constraints in Drizzle.
10. Production placeholder-project cleanup.

**Low** (polish, no functional risk)
11. Smarter `VARCHAR` sizing.
12. More realistic dummy JSON data.
13. CSP headers.
14. Client-side prompt-length indicator.
15. `CLAUDE.md` folder-structure correction.

## 7. v0.7.1 Roadmap

### Objectives
1. Close the two data-integrity/UX bugs found in production testing (Critical #1, #2).
2. Give the compiler pipeline a regression safety net before touching it further.
3. Ship the history/navigation feature the backend already supports.
4. Harden the deployment pipeline so this release's five-round production
   debugging cycle can't recur silently.

### Features
- Project display-name resolution in the generator's project selector.
- A generations-history view per project (list, view a past generation's
  artifacts, delete) using the existing repository functions.
- FK-column auto-indexing in the SQL and Drizzle compilers.
- A new analyzer warning (not error — preserves the "AST stays valid,
  compiler still runs" philosophy) for join-table-shaped tables missing a
  composite unique constraint.

### Architecture changes
- No changes to the AST shape, validator, or compiler contract are required
  for the Critical/High items — FK indexing is additive within
  `lib/compiler/sql` and `lib/compiler/drizzle` (new render step consuming
  existing `relationshipsBySourceTable` from `lib/compiler/shared`); the
  join-table warning is additive within `lib/ast/analyzer.ts` (new
  `AnalysisWarningCode` member, following the existing pattern exactly).
- History UI is additive: new Server Actions wrapping the already-existing
  `getProjectGenerations`/`getGeneration`/`deleteGeneration`, no new
  repository code needed.
- No changes to `AIProviderAdapter`, `SchemaCompiler<TOutput>`, or
  `CompilerRegistry` — the provider/compiler patterns are sound and don't
  need touching for this milestone set.

### Folder changes
```
app/dashboard/projects/[projectId]/history/page.tsx   (new — history view)
lib/actions/generation.actions.ts                     (new — history-facing Server Actions)
components/dashboard/generation-history.tsx            (new)
components/dashboard/generation-history-item.tsx        (new)
tests/                                                  (new — or colocated *.test.ts, see Testing strategy)
docs/planning/v0.7.1-roadmap.md                         (this document)
```

### Database changes
None required. FK indexing and the join-table warning only affect
*generated output* (SQL/Drizzle text the user downloads), not this app's
own Supabase schema. History UI reads existing tables through existing
repositories. If `SUPABASE_SERVICE_ROLE_KEY` is ever put to real use, or if
CI-driven migrations are added, `DATABASE_URL` would need to be added to
Vercel — not required for anything in this roadmap.

### Testing strategy
- **Decided (roadmap review, see §11):** test runner is **vitest** — fast,
  ESM-native, works cleanly with this project's existing TypeScript/Zod
  stack. `package.json` currently has zero test infrastructure, so this is
  a green-field choice, not a migration. `node:test` (zero new dependencies,
  built into Node 24) was the considered alternative; vitest's DX and
  tooling/IDE integration won given how many compiler fixtures this suite
  will need to author and maintain.
- **Decided (roadmap review):** compiler tests use **hand-written expected-
  output assertions, not snapshot testing.** Every compiler's entire design
  promise is "byte-identical, deterministic, human-reviewable output"
  (documented in `docs/architecture/sprint5-ast.md`) — a snapshot test would
  just re-encode whatever the compiler currently emits and rubber-stamp any
  future change, including an unintended one, which defeats the purpose of
  having tests protect Milestone 2's compiler changes.
- Unit-test each of the 5 compilers as pure functions: fixture
  `CanonicalSchemaAST` in, exact string out.
- Unit-test `lib/ast/analyzer.ts` per `AnalysisErrorCode`/`AnalysisWarningCode`
  — one fixture AST per check, asserting the exact code fires.
- Integration-test `buildGeneratedArtifacts` against a real `compileAll()`
  output (the function is already exported specifically for this reason,
  per its own doc comment).
- **CI pipeline is part of Milestone 1, not a separate future item**
  (added during roadmap review — a test suite nothing enforces is an
  opt-in safety net, not a real one). A minimal
  `.github/workflows/ci.yml` runs `npm run lint`, `tsc --noEmit`, `npm
  test`, and `next build` on every PR. No such workflow exists in the repo
  today (confirmed: no `.github/` directory at all).
- Do **not** add live-Gemini-call tests to CI — this release's own
  investigation showed Gemini calls are slow, quota-constrained, and
  non-deterministic; live-provider testing stays a manual/pre-release step
  the way it was done this release, not a CI gate.
- **Vercel Preview deployments** (already available — `GEMINI_API_KEY` is
  scoped to Preview *and* Production, currently unused) become the standard
  verification step for Milestone 2 and Milestone 3 changes before
  promoting to production, rather than verifying against production
  directly the way this release's post-hoc debugging had to.

### Risks
- **Compiler changes without tests first is the single biggest regression
  risk** — FK-indexing and any join-table logic touch the SQL/Drizzle
  compilers, which currently have zero automated coverage. Mitigation:
  Milestone 1 (test infra) must land and cover current behavior *before*
  Milestone 2 touches compiler output.
- **History UI surfaces data that was never meant to be browsed before** —
  RLS is the only access control on `generations`; a history view is a new
  *read* pattern, not a new privilege, but it's worth a deliberate check
  that `getProjectGenerations` truly can't cross project/user boundaries
  (it was reviewed and found correct during the Sprint 5 audit, but a new
  UI surface is a new place for a future regression to matter).
- **Vercel Git integration is an operational change, not a code change** —
  low technical risk, but connecting it will make the *next* push to `main`
  auto-deploy, which is a behavior change worth flagging explicitly before
  doing it (same spirit as this release's "ask before deploying" pattern).
- **Gemini quota** — this release's investigation burned through a free-tier
  daily quota multiple times during live testing. Any v0.7.1 work involving
  live-Gemini verification should budget for that constraint explicitly
  rather than assume unlimited test calls.

### Estimated implementation order
1. Test infrastructure + CI (Milestone 1) — nothing else should touch the compilers first.
2. Critical bug fixes (Milestone 2a → 2b → 2c) — project-selector display first (zero risk, ships immediately), then FK indexing, then the join-table warning — each independently branched and reviewed, all covered by Milestone 1's tests as they land.
3. History/navigation UI (Milestone 3) — independent of the compiler work, can run in parallel with Milestone 2 if capacity allows.
4. Deployment/ops hardening (Milestone 4) — **has no code dependency on any other milestone** (pure Vercel/Supabase configuration); sequenced here for focus, not because it's technically blocked — it could be done on day one if preferred.
5. Medium/Low polish (Milestone 5) — enum reconsideration, CHECK-constraint prompt guidance, composite FK in Drizzle, VARCHAR sizing, dummy data realism, CSP, client-side prompt UX.

### Definition of Done for v0.7.1
Decided during roadmap review, to avoid discovering scope disagreements
mid-implementation:
- **Release-blocking:** Milestones 1, 2a, 2b, 2c, 3.
- **Independent, human-gated, not release-blocking:** Milestone 4 — an
  operational decision that can happen before, during, or after the
  code milestones ship, on its own timeline.
- **Not release-blocking, ships incrementally:** Milestone 5 — may spill
  into v0.7.2 without holding up v0.7.1.

## 8. Milestones

### Milestone 1 — Test Infrastructure Foundation
- **Goal:** Add a test runner, CI enforcement, and full coverage of current
  compiler/analyzer behavior, so every later milestone has a regression
  safety net that's actually enforced, not just present.
- **Files affected:** `package.json` (add `vitest`, a `test` script),
  new `lib/compiler/**/*.test.ts` (one per compiler + shared helpers),
  new `lib/ast/analyzer.test.ts`, new `lib/ast/validator.test.ts`,
  new `lib/services/generation.service.test.ts` (targeting
  `buildGeneratedArtifacts` specifically), new
  `.github/workflows/ci.yml`.
- **Dependencies:** None — can start immediately.
- **Acceptance criteria:** `npm test` exists and passes; every
  `AnalysisErrorCode`/`AnalysisWarningCode` has at least one fixture test;
  every compiler has a determinism test (compile twice, assert identical
  output) and a "produces expected output for a known AST" test using
  hand-written assertions (not snapshots — see Testing strategy); CI runs
  lint + typecheck + test + build on every PR and blocks merge on failure.
- **Rollback strategy:** Purely additive (new files, one `package.json`
  script, one workflow file) — revert by deleting the new test files, the
  script entry, and the workflow; zero risk to runtime code.

### Milestone 2 — Critical Bug Fixes

Split into three independent items per roadmap review (§11) — different
files, different risk profiles, no reason to bundle them into one branch/PR.
All three depend on Milestone 1 landing first.

**2a — Project selector shows project title, not raw UUID**
- **Goal:** Fix the visibly-broken UX found in production testing.
- **Files affected:** `components/dashboard/schema-generator.tsx` (Project
  `<Select>` — resolve `project.title` for display, keep `project.id` as
  the value).
- **Dependencies:** None beyond Milestone 1's CI existing to catch a
  regression; this is a pure UI change with no pipeline/compiler risk, so
  it can ship first within Milestone 2 regardless of 2b/2c's progress.
- **Acceptance criteria:** Project selector shows project titles in local
  and Preview-deployment testing; selecting a project still submits the
  correct `project.id`.
- **Rollback strategy:** Single-file, single-concern change — trivial revert.

**2b — FK-column indexing in SQL and Drizzle compilers**
- **Goal:** Every foreign-key column gets a supporting index in generated
  output, closing a standard, well-known Postgres performance gap.
- **Files affected:** `lib/compiler/sql/render-relationship.ts` or a new
  `lib/compiler/sql/render-fk-index.ts` (emit `CREATE INDEX` per FK source
  column set); `lib/compiler/drizzle/render-table.ts` (equivalent index
  emission for Drizzle's `index()` builder).
- **Dependencies:** Milestone 1.
- **Acceptance criteria:** Every relationship's source-column set gets a
  corresponding index in both SQL and Drizzle output for a fixture AST with
  FKs; all Milestone 1 tests still pass plus new tests for this change;
  verified against a Preview deployment generation before promoting.
- **Rollback strategy:** Independently revertable from 2a/2c — if this
  produces unexpected index bloat or naming collisions in real generations,
  revert this alone.

**2c — Join-table composite-uniqueness analyzer warning**
- **Goal:** Flag (warn, not error — AST stays valid, compiler still runs)
  a table shaped like a join table with no uniqueness constraint on its FK
  pair, closing the data-integrity gap observed in a real "Blog" generation.
- **Files affected:** `lib/ast/analyzer.ts` (new
  `AnalysisWarningCode.LikelyJoinTableMissingCompositeKey` or similarly
  named check).
- **Dependencies:** Milestone 1.
- **Acceptance criteria:** A fixture AST shaped like a join table (two FK
  columns, no other required columns, no existing unique constraint)
  produces the new warning; no false positives on non-join-table fixtures
  (verified with at least one fixture that has two FK columns but *does*
  have a legitimate reason not to warn, if one exists, to guard against
  over-triggering); all Milestone 1 tests still pass plus new tests for
  this change.
- **Rollback strategy:** Independently revertable from 2a/2b — single new
  warning code, no change to existing error/warning behavior.

### Milestone 3 — History & Navigation UI
- **Goal:** Surface the already-implemented `getProjectGenerations`/

- **Files affected:** new `app/dashboard/projects/[projectId]/history/page.tsx`,

  `components/layout/app-sidebar.tsx` or `projects-panel.tsx` for navigation

- **Dependencies:** None on Milestones 1/2 technically, but sequenced after

- **Acceptance criteria:** A user can navigate from a project to a list of

  RLS policies, not by adding new application-layer checks; verified
  against a Preview deployment before promoting.
- **Rollback strategy:** Entirely new routes/components/actions — disable by
  removing the navigation entry point (single-line revert) without needing

### Milestone 4 — Deployment & Ops Hardening
- **Goal:** Remove the operational fragility that caused this release's
  multi-round production debugging cycle.
- **Files affected:** none (Vercel project configuration only — `vercel git
  connect`, Supabase dashboard project cleanup); optionally
  `docs/architecture/` gets a short "Deployment" doc describing the manual
  vs. Git-integrated tradeoff and the Sensitive-variable verification
  limitation, for the next person who hits this.
- **Dependencies:** None — independent of all code milestones. Should be
  done with explicit user sign-off given it changes production behavior
  (every future push to `main` would start auto-deploying).
- **Acceptance criteria:** A test push to a non-`main` branch (or a
  deliberate `main` push, by agreement) triggers an automatic Vercel
  deployment; production `projects` table no longer contains the
  "Testing"/"dsasasas"/"sample test" placeholder rows from this release's
  QA.
- **Rollback strategy:** `vercel git disconnect` reverses the integration
  instantly; placeholder-project deletion is a one-way data change and
  should be confirmed with the user before executing, same as any
  production data deletion. **Strengthened during roadmap review:** connect
  Git integration against a non-`main` branch first (or configure a branch
  protection rule on `main`) and verify one full deploy cycle via that path
  before ever letting a `main` push auto-deploy to production — the
  previous approach of "connect it, then disconnect if something goes
  wrong" risks the exact kind of under-verified production deploy this
  release already spent five rounds recovering from.

## 9. Areas to refactor before adding new features

- **Test infrastructure is the one true prerequisite.** Every other
  refactor/feature in this plan touches either the compilers or the
  analyzer, both of which currently have zero automated coverage. This is
  not optional groundwork — it's the difference between "verified" and
  "hoped" for every subsequent change.
- **The project-selector display bug and the future history UI share a
  concern** (resolving a project's display name from its id) — worth
  extracting a single small helper (e.g. a `projectLabel(project)` function
  or simply ensuring both surfaces consume `project.title` directly from the
  same `Project` type already returned by `project.repository.ts`) rather
  than fixing the selector in isolation and re-solving the same problem
  when the history UI is built in Milestone 3.
- **No other structural refactor is warranted right now.** The repository
  pattern, compiler pattern, and provider pattern were all reviewed during
  the v0.7.0 audit and found consistent and sound (every compiler
  implements `SchemaCompiler<TOutput>` uniformly, no special-casing in
  `CompilerRegistry`; every repository follows the same
  `RepositoryResult<T>` shape). Refactoring working, well-reviewed
  abstractions before there's a concrete second use case to justify it would
  be premature.

## 10. Implementation order to minimize regressions

1. **Milestone 1 (tests) first, unconditionally.** Nothing that touches
   `lib/compiler/` or `lib/ast/analyzer.ts` should happen before this exists
   — it's the only thing standing between "we changed the SQL compiler" and
   "we know we didn't silently break it."
2. **Milestone 2 (critical fixes) second**, now protected by Milestone 1's
   tests as they land — smallest-blast-radius fixes first within the
   milestone (project-selector display, a pure UI change with no pipeline
   impact) before the compiler-touching ones (FK indexing, join-table
   warning).
3. **Milestone 3 (history UI) can run in parallel with Milestone 2** if
   there's capacity — it touches an entirely disjoint set of files (new
   routes/components/actions, existing repositories) and shares no risk
   surface with the compiler work.
4. **Milestone 4 (deployment hardening) has no code dependency on anything
   else and can run on its own timeline**, but is explicitly gated on user
   sign-off before flipping Git integration on, since it's a standing
   behavior change to how every future release ships.
5. **Milestone 5 (polish) last**, and incrementally — none of it is
   blocking, all of it benefits from the test infrastructure being in place
   first, and shipping it piecemeal avoids bundling unrelated risk into one
   large change.
