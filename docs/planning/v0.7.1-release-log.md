## Entry 2 — Milestone 1, Phase 1: Vitest Foundation

**Date:** 2026-07-23

**Phase:** D (implementation) — Milestone 1, Phase 1 of 5 (Vitest Foundation)

**What happened:**
Installed `vitest` (the only new dependency), added `test`/`typecheck`
scripts, and wrote a minimal `vitest.config.ts` — every field justified by
empirically removing it and re-running `npm test`, not by assumption.
`resolve.tsconfigPaths: true` (Vite's native option) replaced an initial
`vite-tsconfig-paths` plugin dependency once proven equivalent. Added
exactly one temporary smoke test proving the harness itself works, later
relocated to `test/smoke.test.ts` per review. No application code changed;
`lint`/`typecheck`/`build` all confirmed clean before and after.

**Known Issues (recorded, not fixed — see rationale below):**

- **Issue:** Running `npm test` prints `API key should be set when using
  the Gemini API.` to stderr (twice, once per smoke test).
- **Why it happens:** `test/smoke.test.ts`'s second test imports
  `lib/services/generation.service.ts`, which transitively imports
  `lib/ai/client.ts`. That file constructs
  `new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY })` at module scope
  — it runs on import, not on use. Vitest does not load `.env.local` (only
  Next.js's own `dev`/`build` commands do that automatically), so
  `GEMINI_API_KEY` is `undefined` in the test process, and the `@google/genai`
  SDK logs this warning the moment the module is evaluated.
- **Why it's harmless:** the warning is a `console`-level notice from the
  SDK's own constructor, not a thrown error — both smoke tests still pass.
  Nothing in Phase 1 (or Phase 4, when it comes) actually *calls* the
  Gemini API; the assertion only checks that `buildGeneratedArtifacts` is
  a function, proving the module loaded, not that Gemini was reachable.
- **Why it's deferred to Phase 4, not fixed now:** loading environment
  variables into the test process isn't one of the four things Phase 1 was
  explicitly scoped to configure (Node environment, `@/*` alias,
  `react-server` condition, TypeScript support) — adding it now would be
  exactly the kind of "future-facing option not required for this phase"
  the architecture review asked to avoid. Phase 4 is where
  `generation.service.test.ts` will exercise this same import path for
  real coverage, and that's the natural place to decide whether the noise
  is worth silencing (e.g. mocking `lib/ai/client.ts`, or accepting the
  warning as expected output in a test environment) rather than deciding
  it preemptively here.

**Outcome:** Phase 1 verification complete (`npm test`, `npm run lint`,
`npm run typecheck`, `npm run build` all clean). Awaiting approval before
Phase 2 (AST & Analyzer Tests).
