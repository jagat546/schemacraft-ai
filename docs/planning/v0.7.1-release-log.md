## Entry 2 — Milestone 1, Phase 1: Vitest Foundation

**Date:** 2026-07-23

**Phase:** D (implementation) — Milestone 1, Phase 1 of 5 (Vitest Foundation)

**What happened:**
Installed `vitest` (the only new dependency), added `test`/`typecheck`
scripts, and wrote a minimal `vitest.config.ts` — every field justified by
empirically removing it and re-running `npm test`, not by assumption.
`resolve.tsconfigPaths: true` (Vite's native option) replaced an initial
`vite-tsconfig-paths` plugin dependency once proven equivalent. Added
exactly one temporary smoke test proving the harness itself works, later
relocated to `test/smoke.test.ts` per review. No application code changed;
`lint`/`typecheck`/`build` all confirmed clean before and after.

**Known Issues (recorded, not fixed — see rationale below):**

- **Issue:** Running `npm test` prints `API key should be set when using
  the Gemini API.` to stderr (twice, once per smoke test).
- **Why it happens:** `test/smoke.test.ts`'s second test imports
  `lib/services/generation.service.ts`, which transitively imports
  `lib/ai/client.ts`. That file constructs
  `new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY })` at module scope
  — it runs on import, not on use. Vitest does not load `.env.local` (only
  Next.js's own `dev`/`build` commands do that automatically), so
  `GEMINI_API_KEY` is `undefined` in the test process, and the `@google/genai`
  SDK logs this warning the moment the module is evaluated.
- **Why it's harmless:** the warning is a `console`-level notice from the
  SDK's own constructor, not a thrown error — both smoke tests still pass.
  Nothing in Phase 1 (or Phase 4, when it comes) actually *calls* the
  Gemini API; the assertion only checks that `buildGeneratedArtifacts` is
  a function, proving the module loaded, not that Gemini was reachable.
- **Why it's deferred to Phase 4, not fixed now:** loading environment
  variables into the test process isn't one of the four things Phase 1 was
  explicitly scoped to configure (Node environment, `@/*` alias,
  `react-server` condition, TypeScript support) — adding it now would be
  exactly the kind of "future-facing option not required for this phase"
  the architecture review asked to avoid. Phase 4 is where
  `generation.service.test.ts` will exercise this same import path for
  real coverage, and that's the natural place to decide whether the noise
  is worth silencing (e.g. mocking `lib/ai/client.ts`, or accepting the
  warning as expected output in a test environment) rather than deciding
  it preemptively here.

**Outcome:** Phase 1 verification complete (`npm test`, `npm run lint`,
`npm run typecheck`, `npm run build` all clean). Awaiting approval before
Phase 2 (AST & Analyzer Tests).

## Entry 4 — Milestone 1, Phase 3: Compiler Tests

**Date:** 2026-07-23

**Phase:** D (implementation) — Milestone 1, Phase 3 of 5 (Compiler Tests)

**What happened:**
Read every line of all 5 compilers' source (`lib/compiler/{sql,drizzle,json,
markdown,mermaid}` plus the shared helpers) fresh before writing anything —
no assertion was written from memory of what a compiler "probably" does.
For SQL, Drizzle, Markdown, and Mermaid, expected output strings were
computed entirely by hand from the actual rendering logic, then verified
by running the tests; one hand-tracing error was found and corrected (see
below). For JSON, the FNV-1a-hashed UUID placeholder values were
deliberately *not* hand-computed — verifying a 32-bit hash by hand isn't a
reliable way to catch a real regression — and the tests instead assert
UUID shape (regex) plus cross-row relational correctness (the same value
appears in both the FK column and the row it should reference), which is
a more meaningful test of intent than a hardcoded hash string would be.

**One genuine correction found during verification, not assumed away:**
In the Drizzle composite-primary-key test, I initially expected
`post_id`/`tag_id` to render `.notNull()`. The actual (correct) behavior
is `.primaryKey()` on each — `renderColumnProperty` checks membership in
the table's *resolved* primary key set regardless of whether the PK was
declared at the table level or the column level, so a column that's part
of a table-level composite PK still gets its own `.primaryKey()` chain
call. This was my error in tracing the code, not a bug — fixed the test
expectation, not the compiler.

**Compiler coverage summary (138 total tests across 8 files, up from 47).**
One regression test per compiler except JSON (two) — see the dedicated
list below for what each one protects and why; not repeated per-bullet
here.
- **SQL (23 tests):** happy path (multi-table, index, FK with two actions);
  determinism; empty index/FK sections produce no stray blank lines;
  identifier/string-literal escaping (`"`, `'`); all 5 literal JS types;
  all 5 foreign key actions; auto-generated constraint naming;
  declaration-order independence for FKs; table-level UNIQUE/CHECK;
  `CREATE UNIQUE INDEX`; composite PK resolution; 1 regression test.
- **Drizzle (15 tests):** happy path (2-arg form, single-column FK with
  `onDelete`, `relations()` blocks); determinism; 3-arg form for composite
  PK / table-level unique / index; the `customType` bytea helper (emitted
  once, only when needed); enum-as-`text()` with a values comment;
  expression defaults via `sql\`\`\`` and the resulting `sql` import;
  composite relationships get `relations()` only, never a physical
  `.references()`; duplicate relation key disambiguation; relations-block
  order independence; camelCase conversion; 1 regression test.
- **JSON (18 tests):** happy path with real FK cross-row verification;
  determinism; fixed 3-row count; all 6 string-placeholder heuristics;
  every column type's placeholder formula (int/bigint, float/decimal,
  boolean, date/timestamp, json, binary, enum cycling); literal defaults
  override the "last nullable row is null" rule; 1 regression test.
- **Markdown (16 tests):** happy path (comment, table, indexes,
  constraints, relationships); determinism; singular/plural boundary;
  omitted subsections when empty; slugified ToC anchors; relationship
  name/actions rendering; multi-column relationship formatting; every
  `formatDefaultForDocs` variant; 1 regression test.
- **Mermaid (21 tests):** happy path (PK/UK tags, inferred one-to-many
  cardinality); determinism; one-to-one cardinality; custom relationship
  labels; PK-beats-FK-beats-UK tag priority (both directions); every
  `mapToMermaidType` mapping; identifier sanitization; 1 regression test.

**Regression tests — what each protects and why (architectural rationale,
stated once here rather than per compiler above). All four tie to a real,
identifiable incident or a documented design concern from this project's
history, not a synthetic worst-case:**
1. **SQL, Drizzle, Markdown** — expression/check content renders verbatim,
   unsanitized. Protects the compiler-layer half of a boundary this
   project actually crossed once: the real `UNSAFE_EXPRESSION` incident
   established that content validation is the analyzer's job
   (`lib/ast/analyzer.ts`), and no compiler should quietly grow its own
   redundant (and possibly inconsistent) validation logic.
2. **SQL, Drizzle** — a self-referencing FK (`comments.parent_id ->
   comments.id`) compiles correctly. This is the exact shape a real
   "Blog" AI generation produced. For Drizzle specifically, it's also the
   precise hazard `render-relations.ts`'s own source comment names
   (`relations()` blocks must be emitted after every `pgTable` const, or a
   self-reference would need a not-yet-declared binding) — this test is
   what actually enforces that guarantee holds, not just documents it.
3. **JSON** — two FK columns on one join table resolve independently
   against their own targets. Protects the two-pass `apply-relationships`
   design from a specific failure mode: one relationship's overwrite
   bleeding into another's column. Same real `post_tags`-style shape from
   that "Blog" generation.
4. **Mermaid** — a dangling FK reference (source table not found) is
   omitted, not thrown. Protects the analyzer/compiler validation
   boundary from creeping the other direction: the compiler staying
   defensive about input the analyzer should have already rejected,
   rather than assuming it always will be.

**Production code changes:** none. Every file touched this phase is a new
`*.test.ts` file.

## Entry 6 — Milestone 1, Phase 4: Generation Service Integration Tests

**Date:** 2026-07-23

**What happened:**
Added `lib/services/generation.service.test.ts` (11 tests), scoped
exclusively to `buildGeneratedArtifacts` — the exported, pure mapping
function between `CompileAllResult[]` and `GeneratedSchema` — per its own

this-file-only shared setup, since deliberately injecting `{ ok: false }`

compiler was never registered" (`byId.get(id)` returns `undefined`) are
different branches in `requireCompilerOutput`, with different error
messages ("X compilation failed: ..." vs "X compiler is not registered.").
Both are tested as genuinely distinct cases, not treated as one.

merely *absent* from the input array (`documentation?.ok` evaluating

## Entry 7 — Milestone 1, Phase 5: GitHub Actions CI

**Date:** 2026-07-23

**Phase:** D (implementation) — Milestone 1, Phase 5 of 5 (final phase —
GitHub Actions CI)

**What happened:**
Added `.github/workflows/ci.yml`: checkout → setup Node 24 with npm
caching → `npm ci` → lint → typecheck → test → build, triggered on
`push`/`pull_request` to `main`. No matrices, no other OS, no deploy step,
no coverage upload, no badges — exactly the minimal shape requested.

Two things resolved empirically rather than assumed, both changing what
would otherwise have been guesswork carried over from earlier phases:

1. **Node version.** `package.json` has no `engines` field; its
   `@types/node` devDependency is pinned to `^20`, which would be the
   naive answer — but that's a stale type-only pin, not a runtime
   constraint. The actual authoritative signals both point elsewhere:
   this project's own Vercel production deployment is configured for
   Node.js 24.x (confirmed via `vercel project inspect` during an earlier
   audit), and the local dev environment runs Node v24.14.0. Selected 24,
   matching both real runtime targets rather than the stale type pin.
2. **Whether GitHub secrets are required for `npm run build` to pass in
   CI.** This was flagged as an open question needing explicit sign-off
   back in Phase A, on the assumption that `next build` might fail
   without `GEMINI_API_KEY`/`NEXT_PUBLIC_SUPABASE_URL`/
   `NEXT_PUBLIC_SUPABASE_ANON_KEY` set, since some of this app's routes
   prerender at build time. Tested directly instead of carrying the
   assumption forward: temporarily moved `.env.local` aside (confirmed
   restored intact immediately after) and ran a real `next build` with
   zero environment variables present. **It succeeded, exit code 0.**
   This resolves the open question — no GitHub repo secrets are needed
   for this workflow to pass, simpler than Phase A anticipated, and
   removes what would have been the one item this phase needed your
   sign-off for.

One deliberate interpretation, stated rather than silently applied: the
task listed "Restore npm cache" as its own pipeline stage, separate from
"Setup Node." Implemented as a single `actions/setup-node@v4` step with
`cache: npm`, which is GitHub's own officially-recommended way to cache
npm dependencies (it handles both restore and save, keyed on
`package-lock.json`) — a separate hand-rolled `actions/cache` step
alongside it would duplicate that behavior, not add anything, and
conflicts with the "keep it intentionally minimal" requirement stated in
the same task.

**YAML validity verified two ways**, not assumed from visual inspection:
a generic parse (`js-yaml`) confirmed syntactic correctness, and
`@action-validator/cli` (validates against the actual GitHub Actions
workflow JSON schema, not just YAML syntax) passed with exit code 0.

**Production code changes:** none.
